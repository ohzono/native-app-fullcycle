# Phase 18: 最終Code Review + セキュリティレビュー

Design修正後のコード品質を最終確認します。
Phase 12 と同じ `code-reviewer` を継続利用して、初回指摘の修正確認まで行います。

加えて、実装が固まったこのタイミングで**セキュリティ専用レビュー**を実施します。

```yaml
SendMessage:
  to: code-reviewer
  message: |
    ## コンテキスト
    - Issue: #{issue番号}
    - スコープ: 差分レビュー（Phase 12以降の変更のみ）
    - 対象コミット: Phase 14（修正対応）およびPhase 17（design修正）

    ## レビュアの構え（必須・出力には漏らさない）
    対象 diff を「別のエンジニアが提出した PR」として三人称で帰属させ、無条件に正しいとは仮定せず欠陥を能動的に探すこと（迎合 = sycophancy 抑制）。中立帰属をデフォルトとし「junior」等の能力 prior は使わない。批判性はコードの欠陥に向け、表現は建設的に保つ。網羅性を上げる構えで評価閾値は下げない。
    **リークガード**: 出力に作者帰属・人格言及（「別のエンジニアが書いた」「ジュニア」「junior」等）を漏らさないこと。詳細は `code-review` skill の「Phase 0」を参照。

    ## 実行指示
    Issue #{issue番号} のPRに対して最終コードレビューを実施してください。

    Phase 12の初回レビュー以降に追加された変更のみを対象にレビューしてください。

    ## 確認観点
    - Design修正によるコード品質への影響
    - 新たなバグや問題が混入していないか
    - Phase 12で指摘した問題の修正確認

    ## スコープ外
    - Phase 12でレビュー済みの部分の再レビューは不要

    各観点の評価を「✅ / ⚠️ / ❌」で出力してください。
```

※ Team未使用時は同内容を `Task (mobiledev-fullcycle:app-reviewer)` で実行する。

**出力**: 最終コードレビューレポート

## 状態ファイルへの書き込み

Phase 18 完了時に、最終レビュー結果を状態ファイルに保存する:

```bash
# .full-cycle-state.json の decisions.reviewHistory 配列に追記する:
# "decisions": {
#   ...,
#   "reviewHistory": [
#     ...(既存のエントリ),
#     {
#       "phase": 18,
#       "round": [reviewLoopCount + 1],
#       "type": "final-review",
#       "designImpact": "✅ / ⚠️ / ❌",
#       "newIssues": "✅ / ⚠️ / ❌",
#       "fixVerification": "✅ / ⚠️ / ❌",
#       "needsFix": true/false
#     }
#   ]
# }
```

**重要**: `reviewHistory` は配列。上書きではなく**追記**すること。ループ回数（`round`）を含めることで、何回目のレビューかを追跡できる。

## セキュリティレビュー（実装確定後の最終チェック）

コードレビュー・デザインレビュー・修正を経て実装が固まったタイミングで、セキュリティ専用の観点でレビューする。Phase 12 のコードレビューにもセキュリティ観点はあるが、ここではより深い専門的チェックを行う。

### 実行方法

```yaml
Task:
  description: "PR #[PR番号] のセキュリティレビュー"
  subagent_type: mobiledev-fullcycle:app-reviewer
  prompt: |
    ## コンテキスト
    - Issue: #{issue番号}
    - PR番号: #[PR番号]
    - スコープ: PR全変更（セキュリティ専用レビュー）

    ## レビュアの構え（必須・出力には漏らさない）
    対象 diff を「別のエンジニアが提出した PR」として三人称で帰属させ、無条件に安全とは仮定せず脆弱性を能動的に探すこと（迎合 = sycophancy 抑制）。中立帰属をデフォルトとし「junior」等の能力 prior は使わない。批判性はコードの欠陥に向け、表現は建設的に保つ。網羅性を上げる構えで評価閾値は下げない（脆弱性が無ければ評価は据え置き）。
    **リークガード**: 出力に作者帰属・人格言及（「別のエンジニアが書いた」「ジュニア」「junior」等）を漏らさないこと。詳細は `code-review` skill の「Phase 0」を参照。

    ## 実行指示
    セキュリティ専門の観点で PR #[PR番号] の全変更をレビューしてください。

    以下の観点でチェック:
    - **OWASP Mobile Top 10**:
      - M1: 不適切なクレデンシャル使用（ハードコードされた鍵・トークン）
      - M2: 不十分なサプライチェーンセキュリティ（依存ライブラリの脆弱性）
      - M3: 安全でない認証/認可
      - M4: 不十分な入力/出力バリデーション（SQLインジェクション、XSS）
      - M5: 安全でない通信（HTTP、証明書ピニング欠如）
      - M6: 不十分なプライバシー制御
      - M7: 不十分なバイナリ保護
      - M8: セキュリティ設定ミス
      - M9: 安全でないデータストレージ（平文保存、ログ出力）
      - M10: 不十分な暗号化
    - **認証・認可**:
      - トークン管理（保存場所、有効期限、リフレッシュ）
      - セッション管理
      - 権限チェック
    - **データ保護**:
      - 個人情報の取り扱い（保存、送信、ログ出力）
      - キーチェーン/EncryptedSharedPreferences使用
      - バックアップ除外設定
    - **ネットワーク**:
      - HTTPS強制（ATS/Network Security Config）
      - 証明書ピニング（必要な場合）
      - APIキーの管理方法
    - **プラットフォーム固有**:
      - iOS: Keychain、ATS、App Sandbox
      - Android: ProGuard/R8、android:exported、ContentProvider

    ## 出力要件
    | 観点 | 評価 | コメント |
    |------|------|----------|
    | [観点名] | ✅/⚠️/❌ | [詳細] |

    **セキュリティリスク**: 高/中/低/なし
    **必須修正**: [脆弱性があれば記載（ファイルパス:行番号 + 修正案）]
```

### セキュリティレビュー結果の取り扱い

| セキュリティリスク | アクション |
|------------------|-----------|
| **高**（脆弱性あり） | Phase 14 に戻り修正必須。TDDで修正（カテゴリA） |
| **中**（改善推奨） | Phase 14 に戻り修正推奨 |
| **低/なし** | 記録のみ → Phase 19 へ |

### 状態ファイルへの書き込み

```bash
# .full-cycle-state.json の decisions.securityReview を設定:
# "decisions": {
#   ...,
#   "securityReview": {
#     "risk": "高/中/低/なし",
#     "findings": ["指摘1", "指摘2"],
#     "owaspChecked": true
#   }
# }
```

## 判定結果に応じた分岐

- コードレビュー全て ✅ かつ セキュリティリスク低/なし → **Phase 19 へ**（Fix to Merge）
- コードレビュー ⚠️/❌ あり、またはセキュリティリスク高/中 → **Phase 14 に戻る**
  - `reviewLoopCount` を確認し、`_schema/phase-flow.yaml` の `loops.codeReview` の上限を超えている場合は `terminalState.kind=loop-exhausted` を記録してユーザーに報告して終了（数値をハードコードしない）

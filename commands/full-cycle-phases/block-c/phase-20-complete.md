## CI待ち → マージ準備完了の報告

最終レビュー完了後、CIの状態を確認します。

**デフォルトの動作は「マージ準備完了の報告」までであり、マージは自動実行しない。**

### 1. CIステータス確認

```bash
# PRのCIチェック状態を確認
gh pr checks [PR番号]
```

**判定**:
- 全チェック ✅: マージ可能 → Step 2 へ
- 一部 ⏳ (pending): CI実行中 → ポーリング待機（下記）
- 一部 ❌ (failure): CI失敗 → ユーザーに報告して終了（マージしない）

**CI 待機のポーリング設定**:
- ポーリング間隔: 30秒
- タイムアウト: 15分
- タイムアウト時: 「CIが15分以内に完了しませんでした」とユーザーに報告して終了

### 2. マージの扱い（実行モード別）

- **インタラクティブ実行（`/full-cycle-dev` / `/full-cycle-review`）**: AskUserQuestion で「PR #[番号] をマージしますか？（マージ後、リモートブランチは削除されます）」と確認する。**ユーザーが承認した場合のみ** Step 2a を実行する。承認が得られない場合・確認手段が使えない場合はマージせず、Step 3 の「マージ準備完了」報告で終了する。
- **自律実行（`/parallel-full-cycle` の background orchestrator）**: マージを実行しない。「マージ準備完了」として PR 一覧の報告で終了する（マージの実行はユーザーに委ねる）。

### 2a. マージ実行（ユーザー承認時のみ）

```bash
# リポジトリの許可マージ方式を取得
REPO=$(gh repo view --json nameWithOwner -q '.nameWithOwner')
MERGE_METHODS=$(gh api repos/$REPO --jq '{squash: .allow_squash_merge, merge: .allow_merge_commit, rebase: .allow_rebase_merge}')

# 優先順位: squash > merge > rebase（利用可能なもの）
# 例: squash が許可されていれば --squash、されていなければ --merge
```

```bash
# マージ実行（判定した方式を使用）
gh pr merge [PR番号] --[squash|merge|rebase] --delete-branch

# マージ後、PRがcloseされたことを確認
gh pr view [PR番号] --json state -q '.state'
```

**注意**:
- `--delete-branch` でリモートブランチを自動削除
- マージ後は `/cleanup` でローカルのworktreeと状態ファイルを削除することを推奨

### 3. 完了メッセージ

**マージを実行した場合**:

```markdown
## フルサイクル開発 完了

**Issue**: #[issue番号]
**PR**: [PR URL]
**ステータス**: ✅ マージ完了

### 次のステップ
- `/cleanup #[issue番号]` でローカルのworktreeと状態ファイルを削除
```

**マージを実行しなかった場合（デフォルト）**:

```markdown
## フルサイクル開発 完了（マージ準備完了）

**Issue**: #[issue番号]
**PR**: [PR URL]
**ステータス**: ✅ マージ準備完了（CI 通過、マージは未実行）

### 次のステップ
- PR の内容を確認し、問題なければマージする: `gh pr merge [PR番号]`
- マージ後 `/cleanup #[issue番号]` でローカルのworktreeと状態ファイルを削除
```
